## What we're building

**DriftGuard** is an AI-powered architectural compliance system, built on
IBM Bob, for the IBM Bob Dev Day Hackathon.

The user writes their team's architectural rules in plain English in a file
called `ARCHITECTURE.md`. Three custom Bob modes operate on that file:

- **DriftGuard Reviewer** — read-only mode that finds violations and
  produces a structured report.
- **DriftGuard Refactor** — applies fixes for violations, with explicit
  user approval at each step.
- **DriftGuard Suggest-Rules** — analyzes the codebase and proposes new
  candidate rules based on patterns it observes. **This is the
  differentiator** — the meta-capability that turns DriftGuard from a
  rule enforcer into a knowledge extractor.

## The problem we're solving

Every codebase has architectural intent — written down once in an ADR or
design doc, then violated within months because no enforcement layer
exists. Existing tools (ArchUnit, dependency-cruiser, NetArchTest,
Structurizr) try to close this gap, but they all require an engineer to
translate human-readable rules into a domain-specific language. That
translation is friction, it goes stale, and it's almost never done by the
architect who wrote the original intent.

DriftGuard removes the translation step. The architect writes the rules
once, in English, and Bob is the enforcement engine. This works because
Bob's strengths — full repo context and intent understanding — are exactly
what English-rule enforcement requires.

## Constraints

- **Bobcoin budget: 40.** Track usage in Settings → General. Don't iterate
  on prompts more than 3-4 times before stepping back to fix the
  underlying issue.
- **Bob IDE is required.** The `bob_sessions/` folder with exported task
  histories is a hard submission requirement.
- **No real secrets in code.** If you need API key examples, use clearly
  labeled placeholders.
- **Demo target language: TypeScript/Node.** Express-style service.

## Behavioral guidance for Bob

- Read this brief first; check `AGENTS.md` if it exists for any extra
  project context.
- `ARCHITECTURE.md` is the source of truth for rules. When generating
  rules, every rule has three parts: **Rule statement**, **Rationale**,
  and **Exception clause**.
- When the prompter runs a phase prompt, do that phase end-to-end 
  unless there are questions and stop.
  Don't run ahead into the next set of work.
- When in doubt, ask before committing significant work. False starts
  cost coins.
- Output should be production-quality — judges score on completeness and
  design as well as innovation.
- The Suggest-Rules mode is the highest-leverage piece of the project
  for hackathon scoring. When you get to it, spend the iteration time it
  needs.

## Hackathon scoring (for context)

Each submission is judged on four 5-point criteria:

1. **Completeness and feasibility** — how complete and well-thought-out.
2. **Creativity and innovation** — how unique the approach is.
3. **Design and usability** — quality of UX and ease of adoption.
4. **Effectiveness and efficiency** — does it address a real, scalable problem.

Every artifact we build should serve at least one of these.
